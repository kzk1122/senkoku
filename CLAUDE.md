# CLAUDE.md − senkoku(線刻)開発ガイド

## プロジェクト概要

**線刻(せんこく / senkoku)** は、Apple Pencil でイラストの基礎を段階的に練習する Web アプリ(PWA)。
ポップなデザイン(2026-07 に和テイストからリブランド。「道場」の世界観は廃止)。どの課題も自由に選べる(解放ゲートなし。70点は合格ライン表示のみ)。

- 開発者はイラストほぼ初心者。自分自身の練習用に開発している
- リポジトリ: https://github.com/kzk1122/senkoku
- 公開先: Cloudflare Pages(静的ホスティング、ビルド設定なし、output directory `/`)

## 全体構想(3ステージ)

1. **Stage 1: 線の練習**(index.html)− 線をなぞって採点。線画・運筆の基礎
2. **Stage 2: かたちの練習**(keitai.html)− 図形分解による人体アタリ(採点あり)、タイマー式ジェスチャードローイング(30秒〜2分、プロシージャル生成ポーズ)。お手本画像のアップロード対応は今後の拡張
3. **Stage 3: 模写の練習**(mosha.html)− モチーフのお手本を「隣に表示 → 薄く下敷き → 記憶模写」の3モードで模写。採点時に差分オーバーレイ(コーラルでお手本を重ねる)
4. 将来構想: Claude API による描画講評機能

## 技術方針(重要・変更しないこと)

- **Vanilla JS / ビルドツールなし**の静的サイト。npm 依存を導入しない
- 開発者の既存プロジェクト(Iemono, HibiOri)と同じ系統: 素の Web 技術 + Cloudflare Pages
- UI は日本語。デザインは「クリーム × コーラル × ティール × イエロー」のフルポップ(css/style.css の CSS 変数がデザイントークン。2026-07 に和テイストからリブランド)
  - `--paper: #FFFDF7` / `--ink: #33323E` / `--shu: #FF6B57`(コーラル=アクセント)/ `--teal: #2EC4B6` / `--sunny: #FFC93C` / `--nibi: #9C99A8`
  - **変数名は旧名(--shu / --nibi)のまま値だけ差し替えている**(JS/CSS の互換のため)。ボタン・チップは黒縁+ズレ影(--pop-shadow)のステッカー風
  - フォント: M PLUS Rounded 1c(500/700/800)のみ
- ロゴ: 小文字「senkoku」+ コーラルの始点● + 一筆の払い(ガイド意匠)。書体は **M PLUS Rounded 1c 800 をアウトライン化した SVG**(`icons/logo.svg` = ヘッダー用ワードマーク、`icons/icon.svg` = PWA アイコン。フォント読み込み不要)。文字を変える場合は同フォントからパスを再生成すること
- 採点結果は「ランクバッジ」(S・A・B・C・D のステッカー風)で表示するのがアプリの象徴的 UI
- データ保存は現状 localStorage(`senkoku_progress_v1`)。**スコア履歴・練習日数を持つ段階で IndexedDB へ移行予定**

## Stage 1 の実装仕様(js/app.js)

### 課題

`COURSES` 配列で定義。各課題は `pathFn(t)` (t∈[0,1] → 正規化座標 0〜1) でお手本パスを生成。
現在28課題(表示名。**課題 id は旧名由来のまま変えない** = 最高記録の互換のため): 直線6(よこ線/←・たて線/↑・ななめ↘/↗)、カーブ5(はらい・ぎゃくはらい・なみ線・たてなみ・おおなみ)、まる系5(まる・まる・ぎゃく・まる3つ・うずまき・うずまき外へ)、大カーブ5(アーチ・Uの字・S字・S字よこ・むげん∞)、筆圧4(キープ・だんだん強く・だんだん弱く・スッと抜く)、ハッチ3(ななめ・たて・クロス12本)。PASS_SCORE(70)は合格ライン「良」の表示に使う(解放ゲートはない)。
筆圧課題(筆圧キープ・だんだん強く・スッと抜く)は `pressureFn(t)` (t → 目標筆圧 0〜1) を持ち、補助線が目標筆圧を太さで可視化する。
複数ストローク課題(ハッチ系・まる3つ)は `hatch: { lines: N }` + `lineFn(i, t)` を持つ。筆は蓄積され N 本目で自動採点(`scoreStrokes` は常に筆のリストを受け取る。単筆課題は要素1つ)。`scoreCurrent` が `partTargets` を渡すため、曲線部品(同心円)の固有曲率は減点されず、ライン抜けは部品最低covゲートで0点になる。

### Apple Pencil 入力

- PointerEvent 使用。`pressure` で線の太さを変える: `lineWidth = 2 + 7 * pressure`(マウスは 0.5 相当)
- `getCoalescedEvents()` で高解像度サンプリング
- パームリジェクション: 一度 `pointerType === "pen"` を検知したら以降 touch を無視(state.penSeen)
- canvas は `touch-action: none`、devicePixelRatio でスケーリング
- 一筆書き方式: pointerdown で前の線をクリア、pointerup で自動採点(8点未満の線は無視)

### 採点アルゴリズム(scoreStroke)

お手本パス 240 点サンプルとの比較。距離はお手本のバウンディングボックス対角線 `size` で正規化。

| 指標 | 重み | 計算 |
|---|---|---|
| 精度 acc | 50% | 描画各点→お手本の平均最短距離。`clamp01(1 - (meanErr - 0.008) / 0.05) * 100` |
| 網羅 cov | 30% | お手本各点の 5.5% 近傍を通過した割合 |
| 滑らか smo | 20% | 前平滑化(step/2 で等間隔リサンプリング → 3点移動平均×2)した線の角度二階差分(step = size/45)から、お手本自身の曲率を引いた余分なガタつき。`clamp01(1 - excess / 0.30) * 100` |

**網羅ゲート**: `total = round((acc*0.5 + cov*0.3 + smo*0.2) * clamp01((cov - 40) / 50))`
→ 線を途中でやめると大幅減点。「一筆で描き切る」ことを最優先で評価する思想。

**ハッチング**: 精度・網羅は全ラインのサンプル集合に対して全筆の点で評価(間隔の乱れ→精度減、線の抜け・重複→網羅減)。滑らかさは1本ずつ測って点数加重平均(基準曲率 0)。網羅ゲートは厳しめの `clamp01((cov - 60) / 35)`(1本抜け cov83% で 60 点前後まで落ち不合格になる)。
検証済み: きれいな6本=95 / 間隔大きく乱れ=74 / 2本重ね(1本抜け)=0(部品最低covゲート)/ 半分の長さ=8 / 同心円きれい=100。

**部品つき課題(partTargets あり = アタリ・模写)**: 部品ごとの網羅率の最低値 `minPartCov` を計測し、ゲート `clamp01((minPartCov - 20) / 40)` を乗算(60+ で減点なし)。合併の網羅率では下がらない「小部品の描き忘れ」「部品の大きさの大間違い」を確実に不合格にする。

**模写(Stage 3)**: モード0/2 は `fitStrokesTo` で正規化してから `accScale 0.08 / covTolRatio 0.055` で採点。
検証済み: 上手な模写=98 / 並=86 / 比率間違え=78 / 部品配置が滅茶苦茶=47 / 口を描き忘れ=0。

**筆圧課題の追加軸**: `pressureFn` がある課題は4軸配分 `acc*0.35 + cov*0.25 + smo*0.15 + prs*0.25`。
prs = 最近傍お手本点の目標筆圧との平均誤差ベースだが、**絶対値比較ではなくオフセット補正つき**:
ユーザーの平均筆圧を目標カーブの平均に合わせてから(補正幅±0.35まで)誤差を測り、「圧の変化の形」を採点する。
`clamp01(1 - (meanPErr - 0.03) / 0.22) * 100`。筆圧の絶対値は個人差が大きく、絶対値比較だと実機で prs=0 に張り付く(2026-07 確認)。
さらに筆圧ゲート `clamp01((prs - 15) / 45)` を網羅ゲートに乗算(prs 60+ で減点なし)。
検証済み: 自然な筆圧が±0.25 ずれた人でも形が合えば prs 99〜100 / 一定圧でサボった「だんだん強く」=total 55 で不合格 / 制御できず=total 60前後で不合格。

検証済みの挙動(この値を壊さないこと): 完璧トレース=100点 / 軽い人間的ブレ・丁寧なゆっくり線=90点台(smo 70〜99)/ 雑な線=smo 0〜20 / 半分で放棄=20点台。
※滑らかさは 2026-07 に実機調整済み: 前平滑化なしだと Apple Pencil のセンサノイズ+手ブレで丁寧な線でも smo=0 に張り付いた。
**採点ロジックを変更したら `node tools/sim/verify.js` で回帰確認すること**(全課題タイプを合成ストロークで採点し期待値と照合。意図した係数変更の場合は出力を確認して EXPECTED を更新)。

ランク: 90+ S / 80+ A / 70+ B(合格)/ 55+ C / 未満 D。

### その他の UI

- 補助線3段階トグル(濃 0.5 / 淡 0.18 / 無 0)。始点●と方向矢印は常時表示
- 筆の太さ3段階トグル(細/中/太)。`lineWidth = base + k * pressure`。設定は localStorage(`senkoku_pen_v1`)に保存。採点には影響しない(表示のみ)
- 採点後に弱点(最低スコアの軸)に応じた一言アドバイス(noteFor)
- 課題レール(chip)に最高得点と解放状態を表示

## Stage 2 の実装仕様(js/keitai.js)

- **アタリ課題**: `ATARI_COURSES`(頭部・胴体・全身・横顔・腕・手・脚・足の8課題)。各課題は `parts`(部品パスのリスト、1部品=1筆)を持つ複数ストローク課題。パスは ellipsePath / linePath / polyPath / rotEllipsePath ヘルパーで定義。**新課題は形状を SVG レンダリングで目視確認してから入れること**(tools 不要、node でポリライン SVG を書き qlmanage で PNG 化)。
  共通採点エンジンに `partTargets` を渡し、筆ごとに最寄り部品の曲率を滑らかさの基準にする(楕円部品の固有曲率を減点しないため)。
  進捗は Stage 1 と同じ `senkoku_progress_v1` に保存(id: at_head / at_torso / at_body)
- **ジェスチャードローイング**: `POSES`(関節角度プリセット8種)から FK(`buildFigure`)で棒人間+図形分解の
  ポーズをプロシージャル描画。ランダムに左右反転+角度ゆらぎ±5°。タイマー 30秒/60秒/2分、時間切れで描画不可。採点なし
- 入力まわりは `makeSurface()`(canvas ごとのストローク管理・パームリジェクション・筆圧描画)に集約

## Stage 3 の実装仕様(js/mosha.js)

- **題材**: `MOTIFS`(三日月 → 富士 → なみもよう → 瓢箪 → うめの花)。単位正方形 0..1 の `parts`(部品パス)で定義し、キャンバスへ等倍フィット。画像ファイル不要
- **3モード**: 隣に表示(0)→ 下敷き(1)→ 記憶(2、10秒見て隠す)。題材・モードとも自由に選べる。進捗 id は `ms_<題材>_<モード>`(モードごとに最高点を保存)
- **採点**: 「見比べる」ボタンで手動採点。モード0/2 は `fitStrokesTo` で描画をお手本枠へ正規化(位置・大きさ不問、accScale 0.08 に緩和)。モード1 はその場で採点
- **差分オーバーレイ**: 採点時にお手本をコーラル(55%)でユーザーの描画位置へ逆変換して重ねる
- 題材の部品は**隣接部品から網羅許容(5.5%)以上離す**こと。近すぎると描き忘れ検出(部品最低covゲート)が効かない(瓢箪の口で実証済み)

## 開発時の注意

- ローカル確認: `python3 -m http.server 8080`(8000 は他の node プロセスと衝突しがち)。iPad からは `http://<MacのIP>:8080`
- Service Worker のキャッシュが効くので、動作確認時はハードリロード or CACHE 名の更新を忘れない
- **Cloudflare Pages は `/xxx.html` → `/xxx` へ 308 リダイレクトする**。SW で `cache.addAll` すると redirected フラグ付きでキャッシュされ、Safari が「Response served by service worker has redirections」でページを開けなくなる。sw.js はクリーンな Response に作り直して両方の URL キーで保存する実装になっている(壊さないこと)
- **`hidden` 属性は author CSS の `display` 指定に負ける**。オーバーレイ等の表示切替に `hidden` を使う場合は css/style.css の `[hidden] { display: none !important; }` が前提(過去にこれが原因で、非表示のはずの結果オーバーレイが canvas を覆いペン入力を奪うバグがあった)
- コミットメッセージ・コード内コメントは日本語で OK
